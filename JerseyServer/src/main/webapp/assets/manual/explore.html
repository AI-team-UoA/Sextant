Explore data in a SPARQL Endpoint

Sextant provides an explore functionality that allows the exploration of a SPARQL endpoint. This feature supports three basic objectives:

  • Visualize the ontology of the data in the endpoint
  • Describe URIs
  • Create layers by applying filters on a class


Visualize the ontology

To use the explore functionality we go to the Explore panel where we can provide the URI of a SPARQL endpoint and the Port it listens to, as shown in the figure below.


When we Connect to the endpoint we get a tree structure of the ontology Classes. Each class is represented with a button where we show the name of the class and the number of sub-classes it has, as shown in the figure below.


When we click on a class button, the class explands and we can view the button that can be used to retrieve all the properties of the class along with their range, the URI of the class and the sub-classes. In the figures below we show an example of these features. When the properties of a class are retrieved, the class button color changes to green, and the button is disabled.



Describe URIs

Each of the URIs that appear in the explore panel can be resolved. When we click on a URI we will get a pop-up that describes the URI and presents all the triples in the endpoint that have this URI in the subject, predicate or object position, as shown in the figure below. Each of the URIs that appear in the describe pop-up are also resolvable and will update the window with the description of the chosen URI. In this manner we can explore our data using the links of the URIs.



Create layers

In some cases we can also use the explore panel to create layers on the map by applying filters to the properties of a class. If a class in the explore panel has a geometry property, then Sextant is able to create layers out of the data of this class. When we chose to retrieve the properties of a class, Sextant looks for properties with the names asWKT, asGML and hasGeometry that are used to represent geometries. If one or more of these properties are available for the class then a new set of buttons appear, as shown in the following figure.


Each property of the class can have a button that allows the creation of filters over this property. The appears under the range URI of a property if that is one of the following:

  • http://www.w3.org/2001/XMLSchema#string
  • http://www.w3.org/2001/XMLSchema#double
  • http://www.w3.org/2001/XMLSchema#float
  • http://www.w3.org/2001/XMLSchema#decimal
  • http://www.w3.org/2001/XMLSchema#integer
  • http://www.w3.org/2001/XMLSchema#boolean
  • http://www.w3.org/2001/XMLSchema#dateTime
  • Geometry property

When we click on of these buttons, a pop-up appears, depending on the range of the property, to allow the creation of filters. Each of these pop-ups can lead to the creation of up-to three filters, but we can click the same again to apply more filters on this property if needed. In the following figures we can see the forms to create spatial filters and numeric filters on properties.


Each filter has three attributes which are explaned below:

  • TYPE: The type of a filter can be either REGULAR or OPTIONAL. Regular filters mean that this filter is a must condition for the instances of the class, whereas optional filters mean that this filter is not obligatory.
  • RULE: The rule attribute defines the operator we will use for the filter.
  • VALUE: The value attribute defines the value against which the operator will be applied.

When a filter is created on a property, the button changes into to indicate that filters are applied on this property. We can view all the filters that are applied on the class by clicking the button and remove all the filters from a class using the button. When we have applied all the desired filters, we can create a layer from this class using the button. We then get a pup-up where we provide a name for the layer and create it on the map.
